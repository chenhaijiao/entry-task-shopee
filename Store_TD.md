# Store 设计文档

## 缓存方案分析（Login / User / List 搜索）

### 1 Login 缓存（token）

- **缓存目标**：保持会话内登录态，支持刷新后仍保持登录。
- **缓存位置**：`sessionStorage`（仅当前 tab 生命周期），避免长期驻留带来的安全风险。
- **生命周期**：
  - 登录成功：写入 token。
  - 刷新页面：token 仍在，`auth model` 会自动拉取用户信息恢复 UI。
  - 关闭 tab：token 自动清空，需要重新登录。
  - 401/403：请求层统一触发未授权处理，清 token 并跳转登录。

### 2 User 数据缓存（Redux）

- **缓存目标**：用于全局展示（头像、用户信息等），避免在多个页面重复请求/传参。
- **缓存位置**：Redux `user` slice（内存态）。
- **恢复策略**：不把 user 直接持久化到 storage；刷新后通过 token 重新 `fetchMe()` 恢复，避免“持久化 user 过期/脏读”。
- **一致性**：
  - token 变化（登录/退出/过期）会驱动 user 清空或重新拉取；
  - 未授权时统一清理，避免 UI 继续展示旧用户。

### 3 List 页面搜索关键词/筛选缓存（Redux）

- **缓存目标**：列表页与筛选抽屉之间共享筛选条件与展示文案。
- **缓存位置**：Redux `search` slice（内存态）。
- **不做持久化的原因**：筛选条件属于“临时 UI 状态”，持久化容易导致下次打开就带着旧条件，引发困惑。
